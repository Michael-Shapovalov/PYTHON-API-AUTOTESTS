# Новая структура API-тестов (просто и подробно)

Этот файл объясняет, как теперь устроены тесты после перехода на паттерн:

- `API Client` (или Service Object)
- `Steps`
- `Tests`

Также здесь описано, как живут отдельные наборы `smoke` и `full/regression`.

---

## 1) Главная идея

Раньше HTTP-вызовы и логика проверок были смешаны прямо в тестовых файлах.  
Теперь структура разделена по слоям:

1. `client.py` - только работа с HTTP (requests, URL, headers, timeout)
2. `steps.py` - переиспользуемые шаги и вспомогательные операции
3. `test_*.py` - сами тест-кейсы (читаемые сценарии и asserts)

Это дает:

- меньше дублирования
- проще поддержка
- проще расширять тесты
- проще выделять smoke/full наборы через маркеры

---

## 3) Что в каждом слое

## `client.py`

Содержит методы конкретного API:

- `create_or_update`
- `create_or_update_many`
- `get_*`
- `post_*`
- `delete`
- `delete_many`

Пример (идея):

```python
class UnitApiClient:
    @staticmethod
    def create_or_update(put_url, payload=None, auth=None, raw_body=None, timeout=30):
        if raw_body is not None:
            return requests.put(put_url, data=raw_body, auth=auth, headers=JSON_HEADERS, timeout=timeout)
        return requests.put(put_url, json=payload, auth=auth, headers=JSON_HEADERS, timeout=timeout)
```

## `steps.py`

Содержит прикладные шаги и helpers:

- генерация payload (`build_unique_*`)
- поиск сущности в `items`
- логирование проверок (`_log_check`)
- обертки над клиентом (`_put_*`, `_get_*`, `_post_*`, `_patch_*`)
- схемы swagger и общие проверки

Пример:

```python
def _put_unit(put_url: str, payload=None, auth=None, raw_body=None, timeout=30):
    return UnitApiClient.create_or_update(
        put_url, payload=payload, auth=auth, raw_body=raw_body, timeout=timeout
    )
```

## `test_*.py`

Только тестовые сценарии и assertions.
Тесты вызывают шаги, а не `requests` напрямую.

Пример:

```python
def test_unit_get_success_structure(unit_created_for_get):
    get_url, admin_auth, payload = unit_created_for_get
    response = _get_unit_response(get_url, admin_auth)
    assert response.status_code == 200
    data = response.json()
    assert "items" in data
```

## 4) Как соотносятся smoke и full наборы

Важно: `smoke` и `full` - это не разные архитектуры.  
Архитектура одна (`client + steps + tests`), а наборы различаются маркерами и целями.

- `smoke`: короткие критические сценарии работоспособности
- `regression/full`: расширенный набор позитивных/негативных/граничных проверок

Smoke оставлены в отдельных папках:

- `tests/api/unit/smoke/`
- `tests/api/unitgroup/smoke/`
- `tests/api/item/smoke/`

---

## 5) Маркеры

Используются маркеры из `pytest.ini`:

- `smoke`
- `regression`
- `fast`
- `slow`
- `negative`

Принцип:

- запуск `smoke` не зависит от `regression`
- запуск `regression` не обязан включать `smoke`
- можно комбинировать условия через `-m`

---

## 6) Команды запуска

Из папки `tests`:

```bash
pytest api -m smoke -v -s
pytest api -m regression -v -s
pytest api -m fast -v -s
pytest api -m slow -v -s
pytest api -m negative -v -s
```

Smoke в 2 потока (с сохранением модульной целостности сценариев):

```bash
pytest api -m smoke -n 2 --dist loadscope -v -s
```

Почему `--dist loadscope`:

- тесты одного smoke-модуля выполняются в одном воркере
- последовательные smoke-flow не "разъезжаются" между потоками

---

## 7) Обратная совместимость

На переходный период сохранена совместимость:

- старые helper-имена и импортируемые функции не ломаются
- часть старых точек работает как thin-wrapper к новому `steps/client`

Это позволяет мигрировать постепенно, без резкого переписывания всего за один шаг.

---

## 8) Как писать новые тесты дальше

Рекомендуемый порядок:

1. Добавить/расширить метод в `client.py` (если новый endpoint)
2. Добавить шаг в `steps.py` (переиспользуемая логика)
3. Написать тест в `test_*.py` с нужными маркерами
4. Если нужен короткий бизнес-критичный flow - добавить тест в `smoke/`

Короткое правило:

- HTTP-детали -> `client`
- повторяемая логика -> `steps`
- сценарий/проверка -> `test`
