## Подробные тест-кейсы

### ТК-1. Создание единицы измерения с полным набором полей (PUT + GET + сравнение)

**Предусловия:** API доступен, .env настроен.

**Шаги:**
1. Сформировать payload с полным набором полей (integrationId, code, name, acronym, description, internationalAcronym, isDivisible, precision и т.д.) с уникальными значениями.
2. Выполнить PUT /Unit/CreateOrUpdate с Basic Auth.
3. Проверить код 200, наличие results, added=1 или updated=1.
4. Проверить ответ по схеме RestResponseDtoOfRestUnit.
5. Выполнить GET /Unit, найти единицу по integrationId (или acronym).
6. Сравнить поля созданной единицы с отправленным payload.

**Ожидаемый результат:** 200, добавление или обновление, схема соблюдена, единица есть в GET и данные совпадают.

---

### ТК-2. Создание единицы с минимальным набором полей (только name и acronym)

**Шаги:**
1. Отправить PUT /Unit/CreateOrUpdate с полями: name, acronym (уникальные), без integrationId и остальных опциональных.
2. Проверить 200, added=1 или updated=1, наличие единицы в GET.

**Ожидаемый результат:** Успешное создание без ошибок.

---

### ТК-3. Обязательность: отсутствие name — ошибка

**Шаги:**
1. Payload без name (есть acronym и при необходимости другие поля).
2. PUT /Unit/CreateOrUpdate.

**Ожидаемый результат:** HTTP 400 или 200 с ошибкой в results (не added=1).

---

### ТК-4. Обязательность: отсутствие acronym — ошибка

**Шаги:**
1. Payload без acronym (есть name и др.).
2. PUT /Unit/CreateOrUpdate.

**Ожидаемый результат:** HTTP 400 или ошибка в results.

---

### ТК-5. Уникальность: повтор по integrationId — обновление (updated=1)

**Шаги:**
1. Создать единицу с уникальным integrationId (первый PUT → added=1).
2. Второй PUT с тем же integrationId и другим name (и при необходимости другими полями).
3. Проверить updated=1 и что в GET у единицы новое name.

**Ожидаемый результат:** Второй ответ: 200, updated=1; в GET — обновлённое имя.

---

### ТК-6. Уникальность: повтор по code+name — обновление (updated=1)

**Шаги:**
1. Создать единицу по code+name (без integrationId) → added=1.
2. Второй PUT с теми же code и name, изменить description или другие поля.
3. Проверить updated=1 и что в GET данные обновлены.

**Ожидаемый результат:** 200, updated=1.

---

### ТК-7. Тип: name не строка → 400

**Шаги:**
1. Payload с name=12345 (число) вместо строки, остальные поля валидные и уникальные.
2. PUT /Unit/CreateOrUpdate.

**Ожидаемый результат:** 400 или ошибка в results.

### ТК-8. Границы: code/acronym длина 30, name длина 250 — успех

**Шаги:**

2. PUT /Unit/CreateOrUpdate.
3. Проверить 200 и наличие единицы в GET.

**Ожидаемый результат:** 200, успешное создание.

### ТК-14. Идемпотентность: один и тот же запрос дважды — added=1, затем updated=1

3. Второй ответ: updated=1.
4. В GET одна запись с данными из payload.

**Ожидаемый результат:** Первый PUT — added=1, второй — updated=1; в списке одна единица.
