# Проверки PUT /Unit/CreateOrUpdateMany

| № | Название проверки | Что проверяется | Ожидаемый результат |
|---|-------------------|-----------------|---------------------|
| 1 | Успешный ответ 200 и структура (один элемент) | Код 200, поля count, added, updated, deleted, errors, results; один элемент в results без errorMessage | 200, все поля присутствуют, added >= 1, results.length === 1, без ошибки в results[0] |
| 2 | Соответствие ответа схеме RestResponseDtoOfRestUnit | Валидация ответа по jsonschema | Ответ проходит jsonschema.validate |
| 3 | Пустой массив [] — 200 | Тело [] | 200, count=0 или без падения |
| 4 | Несколько элементов (2–3) | Массив из 3 единиц | 200, 3 элемента в results, added+updated >= 1 |
| 5 | Запрос без авторизации — 401 | PUT без auth | 401 |
| 6 | Тело не массив — 400 | Один объект вместо массива | 400 или ошибка в ответе |
| 7 | Невалидный JSON — 400 | Обрезанный JSON | 400, 415 или 422 |

---

## Подробные тест-кейсы

### ТК-1. Успешный ответ 200 и структура (один элемент)

**Предусловия:** API доступен.

**Шаги:**
1. Сформировать массив из одного элемента с уникальными code, name, acronym, integrationId.
2. Выполнить PUT /Unit/CreateOrUpdateMany с Basic Auth.
3. Проверить код 200 и наличие полей: count, added, updated, deleted, errors, results.
4. Проверить, что added >= 1, в results ровно один элемент, у него нет errorMessage (или он пустой).

**Ожидаемый результат:** 200; все перечисленные поля присутствуют; один результат без ошибки.

---

### ТК-2. Соответствие ответа схеме RestResponseDtoOfRestUnit

**Шаги:**
1. Отправить PUT /Unit/CreateOrUpdateMany с массивом из одной валидной единицы.
2. Загрузить схему RestResponseDtoOfRestUnit из swagger.
3. Выполнить jsonschema.validate(instance=response.json(), schema=schema).

**Ожидаемый результат:** 200; валидация проходит без исключения.

---

### ТК-3. Пустой массив [] — 200

**Шаги:**
1. Отправить PUT /Unit/CreateOrUpdateMany с телом [] и Basic Auth.
2. Проверить код ответа и что count=0 или поле отсутствует/допустимо.

**Ожидаемый результат:** HTTP 200; сервер не возвращает ошибку.

---

### ТК-4. Несколько элементов (2–3)

**Шаги:**
1. Сформировать массив из 3 единиц с разными code/name/acronym/integrationId.
2. PUT /Unit/CreateOrUpdateMany.
3. Проверить 200, что в results ровно 3 элемента, сумма added+updated >= 1.

**Ожидаемый результат:** 200; results содержит 3 элемента; хотя бы одна операция добавления или обновления.

---

### ТК-5. Запрос без авторизации — 401

**Шаги:**
1. Отправить PUT /Unit/CreateOrUpdateMany с массивом из одной единицы без заголовка Authorization.

**Ожидаемый результат:** HTTP 401 Unauthorized.

---

### ТК-6. Тело не массив — 400

**Шаги:**
1. Отправить PUT /Unit/CreateOrUpdateMany с телом в виде одного объекта (без обёртки в массив).

**Ожидаемый результат:** HTTP 400 или 200 с ошибкой в results/errors.

---

### ТК-7. Невалидный JSON — 400

**Шаги:**
1. Отправить PUT /Unit/CreateOrUpdateMany с телом в виде невалидного JSON (например, `[{"acronym": "x", "code": "y"`).

**Ожидаемый результат:** HTTP 400, 415 или 422.
