# Интеграция автотестов и документации с Test IT TMS

Этот файл содержит сводку по интеграции pytest-автотестов проекта с Test IT. Когда вернётесь к настройке интеграции — откройте этот документ и попросите ассистента «посмотреть docs/testit_integration.md и помочь с интеграцией Test IT».

---

## 1. Три способа интеграции (pytest → Test IT)

### 1.1. Адаптер pytest (рекомендуется)

- **Пакет:** `testit-adapter-pytest`  
  Установка: `pip install testit-adapter-pytest`
- Подключается к pytest как плагин и при каждом запуске тестов отправляет в Test IT:
  - результаты прогонов (passed/failed и т.д.),
  - метаданные: шаги, сообщения, вложения, ссылки, описание, название, лейблы,
  - привязку к ручным тест-кейсам через `workItemIds`.

**Параметры настройки** (переменные окружения или конфиг адаптера):

| Параметр | Описание |
|----------|----------|
| `TMS_URL` | Адрес инстанса Test IT (например `https://your-company.testit.software`) |
| `TMS_PRIVATE_TOKEN` | API-токен из профиля пользователя в Test IT |
| `TMS_PROJECT_ID` | ID проекта в Test IT |
| `TMS_CONFIGURATION_ID` | ID конфигурации (окружения) |
| `TMS_TEST_RUN_ID` | (опционально) ID существующего прогона для записи результатов |
| `TMS_TEST_RUN_NAME` | (опционально) Имя прогона при автосоздании |
| `TMS_ADAPTER_MODE` | Режим адаптера (0, 1, 2) — фильтрация тестов и создание прогонов |
| `TMS_AUTOMATIC_CREATION_TEST_CASES` | Автосоздание тест-кейсов в Test IT при первом запуске |
| `TMS_IMPORT_REALTIME` | Отправка результатов в реальном времени |
| `TMS_CERT_VALIDATION` | Проверка SSL-сертификата |
| `TMS_PROXY` | Прокси при обращении к Test IT |

**Документация:**

- Pytest: https://docs.testit.software/user-guide/integrations/automation/pytest.html  
- Установка и настройка адаптера: https://github.com/testit-tms/adapters-python/tree/main/testit-adapter-pytest  

**Метаданные адаптера** (маркеры/атрибуты в тестах):

| Метаданное | Назначение |
|------------|------------|
| `testit.displayName` | Имя автотеста в Test IT |
| `testit.title` | Заголовок в карточке автотеста |
| `testit.description` | Описание автотеста |
| `testit.workItemIds` | Массив ID ручных тест-кейсов в Test IT — связь автотеста с документацией |
| `testit.externalId` | Внешний уникальный ID автотеста (например из своей БД тестов) |
| `testit.step` | Шаг внутри теста (отображается в TMS) |
| `testit.addLinks` | Ссылки к результату |
| `testit.addAttachments` | Вложения к результату |
| `testit.addMessage` | Сообщения в результат |
| `testit.className` | Субдиректория автотеста в Test IT (по умолчанию — имя файла) |
| `testit.nameSpace` | Расположение автотеста (по умолчанию — имя директории) |
| `testit.labels` | Лейблы в карточке автотеста |

### 1.2. CLI + JUnit XML

- Запуск тестов с выгрузкой JUnit:  
  `pytest --junit-xml=reports/results.xml ...`
- Установка CLI:  
  `pip install testit-cli`
- Загрузка результатов в Test IT:  
  ```bash
  export TMS_TOKEN=<YOUR_TOKEN>
  testit results import \
    --url <TMS_URL> \
    --project-id <PROJECT_ID> \
    --configuration-id <CONFIGURATION_ID> \
    --testrun-name "Pytest test run" \
    --results reports
  ```

**Плюсы:** минимум зависимостей, удобно для CI.  
**Минусы:** в Test IT попадает только результат прогона (passed/failed), без богатых метаданных и без привязки к тест-кейсам по ID.

**Документация:**  
https://docs.testit.software/user-guide/integrations/cli.html  

---

### 1.3. Public API / API-клиент Python

- **Репозиторий:** https://github.com/testit-tms/api-client-python  
- Позволяет программно:
  - создавать и обновлять автотесты и тест-кейсы,
  - создавать прогоны и загружать результаты,
  - привязывать автотесты к ручным тестам.

Удобно для кастомной логики: например, скрипт импорта описания тестов из своей документации (например из `docs/item/checks_smoke.md`) и создание/обновление сущностей в Test IT; прогоны можно заливать тем же клиентом или через адаптер/CLI.

**Документация:**

- Работа с API: https://docs.testit.software/user-guide/work-with-api.html  
- API-клиенты: https://docs.testit.software/user-guide/integrations/api-clients.html  

Авторизация в API: заголовок `PrivateToken <API Secret Key>` (токен из профиля пользователя Test IT).

## 2. Интеграция документации и передача данных о тестах и прогонах

### Документация по готовым тестам в TMS

- В Test IT создаются ручные тест-кейсы (или импорт из другой системы: https://docs.testit.software/user-guide/migration/importer.html ).
- В коде автотестов через адаптер задаётся **`testit.workItemIds`** — ID этих тест-кейсов. Тогда при каждом прогоне результаты автотеста отображаются в карточке соответствующего тест-кейса.
- Дополнительно можно заполнять **`testit.title`** / **`testit.description`** из внутренней документации (например из `docs/item/checks_smoke.md`), чтобы в Test IT было понятное описание.

### Новые тесты

- **С адаптером:** новые pytest-тесты при первом запуске можно автоматически создавать в Test IT (`TMS_AUTOMATIC_CREATION_TEST_CASES`), либо создавать их скриптом через API-клиент и связывать по `externalId` / `workItemIds`.
- **Через API:** можно массово создавать автотесты из своей структуры (например по списку из markdown/конфига).

### Прогоны автотестов

- **Адаптер:** прогон создаётся или обновляется автоматически при запуске pytest с подключённым плагином; все результаты уходят в Test IT.
- **CLI:** после `pytest --junit-xml=...` загружаете каталог с `results` через `testit results import` — в TMS появится один тест-ран с результатами.
- **API:** создаёте тест-ран и по API отправляете результаты каждого теста.

## 3. Практические шаги для данного проекта

1. **Установить и настроить адаптер**  
   `pip install testit-adapter-pytest`, задать настройки через:
   - переменные окружения `TMS_URL`, `TMS_PRIVATE_TOKEN`, `TMS_PROJECT_ID`, `TMS_CONFIGURATION_ID`, или
   - файл `connection_config.ini` в корне проекта (рекомендуется для локального запуска).

2. **В Test IT** завести проект, конфигурацию и при необходимости ручные тест-кейсы (по `docs/item/checks_smoke.md` и др.).

3. **В коде** для важных тестов задать `testit.displayName` / `testit.title` и по возможности **`testit.workItemIds`** (ID тест-кейсов из Test IT), чтобы документация и автотесты были связаны.

4. **Запускать** прогон с включением адаптера:  
   `pytest api -m smoke --testit ...` или `pytest api -m regression --testit ...` — адаптер отправит в Test IT данные о прогоне и результатах.  
   **Важно:** без флага `--testit` плагин не активируется.

5. При необходимости **автоматизировать создание тест-кейсов** из своей документации — написать скрипт на **api-client-python**, который читает описание тестов и создаёт/обновляет сущности в Test IT.
