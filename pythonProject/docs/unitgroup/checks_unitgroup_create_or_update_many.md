# Таблица проверок PUT /UnitGroup/CreateOrUpdateMany

| № | Название проверки | Что проверяется | Ожидаемый результат |
|---|-------------------|-----------------|---------------------|
| 1 | Успешный ответ 200 и структура (один элемент) | Массив из одной группы с baseUnit | 200, count/added/updated/deleted/errors/results; один элемент в results без errorMessage |
| 2 | Соответствие ответа схеме RestResponseDtoOfRestUnitGroup | Валидация ответа по jsonschema | Ответ проходит jsonschema.validate |
| 3 | Пустой массив [] — 200, без падения | Тело [] | 200, count=0 или отсутствует |
| 4 | Несколько элементов (2–3) — 200, все в results | Массив из 3 групп | 200, в results 3 элемента, added+updated >= 1 |
| 5 | Запрос без авторизации — 401 | PUT без Basic Auth | 401 Unauthorized |
| 6 | Тело не массив — 400 | Один объект вместо массива | 400 или ошибка в ответе |
| 7 | Невалидный JSON — 400 | Обрезанный JSON в теле | 400, 415 или 422 |

---

## Подробные тест-кейсы

### ТК-1. Успешный ответ 200 и структура (один элемент)

**Предусловия:** Создана одна Unit для baseUnit.

**Шаги:**
1. Сформировать массив из одного элемента: [ { code, name, integrationId, baseUnit } ] с уникальными значениями.
2. Выполнить PUT /UnitGroup/CreateOrUpdateMany с Basic Auth.
3. Проверить код 200 и наличие полей: count, added, updated, deleted, errors, results.
4. Проверить, что added >= 1, в results ровно один элемент, у него нет errorMessage (или он пустой).

**Ожидаемый результат:** 200; все перечисленные поля присутствуют; один результат без ошибки.

---

### ТК-2. Соответствие ответа схеме RestResponseDtoOfRestUnitGroup

**Шаги:**
1. Отправить PUT /UnitGroup/CreateOrUpdateMany с массивом из одной валидной группы.
2. Загрузить схему RestResponseDtoOfRestUnitGroup из swagger.
3. Выполнить jsonschema.validate(instance=response.json(), schema=schema).

**Ожидаемый результат:** 200; валидация проходит без исключения.

---

### ТК-3. Пустой массив [] — 200, без падения

**Шаги:**
1. Отправить PUT /UnitGroup/CreateOrUpdateMany с телом [] и Basic Auth.
2. Проверить код ответа и что count=0 или поле отсутствует/допустимо.

**Ожидаемый результат:** HTTP 200; сервер не возвращает ошибку; count=0 или аналогично.

---

### ТК-4. Несколько элементов (2–3) — 200, все в results

**Шаги:**
1. Сформировать массив из 3 групп с разными code/name/integrationId и одним baseUnit.
2. PUT /UnitGroup/CreateOrUpdateMany.
3. Проверить 200, что в results ровно 3 элемента, сумма added+updated >= 1.

**Ожидаемый результат:** 200; results содержит 3 элемента; хотя бы одна операция добавления или обновления.

---

### ТК-5. Запрос без авторизации — 401

**Шаги:**
1. Отправить PUT /UnitGroup/CreateOrUpdateMany с массивом из одной группы (baseUnit можно заглушкой) без заголовка Authorization.

**Ожидаемый результат:** HTTP 401 Unauthorized.

---

### ТК-6. Тело не массив — 400

**Шаги:**
1. Отправить PUT /UnitGroup/CreateOrUpdateMany с телом в виде одного объекта (без обёртки в массив).

**Ожидаемый результат:** HTTP 400 или 200 с ошибкой в results/errors.

---

### ТК-7. Невалидный JSON — 400

**Шаги:**
1. Отправить PUT /UnitGroup/CreateOrUpdateMany с телом в виде невалидного JSON (например, `[{"code": "x", "name": `).

**Ожидаемый результат:** HTTP 400, 415 или 422.
