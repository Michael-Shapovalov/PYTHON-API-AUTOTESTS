# Таблица проверок PATCH /UnitGroup/Delete

| № | Название проверки | Что проверяется | Ожидаемый результат |
|---|-------------------|-----------------|---------------------|
| 1 | Успешное удаление по id — 200 и схема | Создать группу, PATCH Delete с её id | 200, RestResponseDto, группы нет в GET |
| 2 | Успешное удаление по integrationId — 200 | Создать группу, удалить по integrationId | 200, deleted или count/results |
| 3 | Запрос без авторизации — 401 | PATCH Delete без Basic Auth | 401 Unauthorized |
| 4 | Удаление несуществующего id | Тело с id 999999999 | 200 с deleted=0 или 404 или 400 |
| 5 | Пустой body — 400/404 | Тело {} | 400, 404 или 422 |
| 6 | Невалидный JSON — 400 | Обрезанный JSON | 400, 415 или 422 |

---

## Подробные тест-кейсы

### ТК-1. Успешное удаление по id — 200 и схема

**Предусловия:** Созданы Unit и UnitGroup; получен id группы из GET /UnitGroup.

**Шаги:**
1. PATCH /UnitGroup/Delete с Basic Auth и телом { id: id_группы }.
2. Проверить код 200, наличие deleted/count/results, соответствие ответа схеме RestResponseDto.
3. GET /UnitGroup — группы с этим id нет в items.

**Ожидаемый результат:** 200; ответ по схеме; группа удалена.

### ТК-2. Успешное удаление по integrationId — 200

**Шаги:**
1. Создать группу с integrationId. PATCH Delete с телом { integrationId } и Basic Auth.
2. Проверить 200 и deleted или count/results.

**Ожидаемый результат:** 200; удаление зафиксировано.

### ТК-3. Запрос без авторизации — 401

**Шаги:** PATCH /UnitGroup/Delete с телом { id: 999999 } без Authorization.

**Ожидаемый результат:** 401 Unauthorized.

### ТК-4. Удаление несуществующего id

**Шаги:** PATCH Delete с телом { id: 999999999 } и Basic Auth.

**Ожидаемый результат:** 200 (deleted=0) или 404 или 400.

### ТК-5. Пустой body — 400/404

**Шаги:** PATCH Delete с телом {} и Basic Auth.

**Ожидаемый результат:** 400, 404 или 422.

### ТК-6. Невалидный JSON — 400

**Шаги:** PATCH Delete с телом в виде невалидного JSON.

**Ожидаемый результат:** 400, 415 или 422.
